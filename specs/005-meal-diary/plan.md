# Implementation Plan: Meal Diary & History View

**Branch**: `005-meal-diary` | **Date**: 2026-05-27 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `/specs/005-meal-diary/spec.md`

## Summary

Add a day-based food diary page that lets authenticated users review all logged meals for any past calendar day, compare daily macro totals against their saved nutrition goals, navigate between days, and delete individual meal log entries via soft-delete. The implementation reads from the existing `MealLog` and `DailyNutritionGoal` entities (Spec 004 and Spec 003 respectively) — no new database migrations are needed. New work is limited to a `DiaryController`, a `DiaryService` with read and soft-delete operations, response DTOs, a frontend API client, and an upgraded `History.jsx` page with `DayNavigator`, `DailySummary`, and `MealGroup` components.

## Technical Context

**Language/Version**: C# with .NET 10 for the backend, JavaScript with React 18 for the frontend

**Primary Dependencies**: ASP.NET Core controllers, JWT bearer authentication, Entity Framework Core with SQL Server, React Router, Axios, Zustand

**Storage**: SQL Server — reads `MealLogs` (Spec 004) and `UserDailyNutritionGoals` (Spec 003); no new migrations required

**Testing**: Solution build verification plus manual smoke flows (load today's diary, navigate to yesterday, delete an entry). No backend unit-test project or frontend test harness — per repository standard.

**Target Platform**: Web application (same as existing — desktop and mobile browsers)

**Project Type**: Web application with separate backend API (modular monolith) and frontend SPA

**Performance Goals**: Diary page loads within 2 seconds; day navigation responds within 1 second; delete operation completes within 1 second.

**Constraints**: All endpoints require authentication; standard `ServiceResponse<T>` envelope on every response; navigation blocked at today (no future dates); editing individual entries is out of scope for v1 (delete and re-log only); no new EF Core migrations.

**Scale/Scope**: Same v1 scale as Spec 004 (~1,000 active users); at most a few dozen diary entries per day per user.
