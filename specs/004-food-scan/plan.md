# Implementation Plan: AI Food Scan

**Branch**: `004-food-scan` | **Date**: 2026-05-26 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `/specs/004-food-scan/spec.md`

## Summary

Add the core differentiator of GymScan: a photo-based food analysis flow that uses the Gemini vision API to identify a meal and return calorie and macro data, checked against a seeded local Egyptian food database for verified accuracy. Users confirm the result and log it to their daily diary, which feeds the dashboard and history features. A manual food search fallback is available when scanning is not practical. The implementation adds three new database entities (`LocalFoodItem`, `FoodScan`, `MealLog`), an `IFoodVisionService` abstraction backed by Gemini, new service and controller layers, and a fully functional scan page on the frontend.

## Technical Context

**Language/Version**: C# with .NET 10 for the backend, JavaScript with React 18 for the frontend

**Primary Dependencies**: ASP.NET Core controllers, JWT bearer authentication, Entity Framework Core, SQL Server provider, FluentValidation, Serilog, `HttpClient` (for Gemini REST API calls), React Router, Axios, Zustand, TanStack React Query

**Storage**: SQL Server for `LocalFoodItem`, `FoodScan`, and `MealLog` entities; local filesystem (`wwwroot/uploads/scans/`) for scan image files

**Testing**: Solution build verification plus manual smoke flows (analyze, log, search, AI-down fallback). No backend unit-test project or frontend test harness — per repository standard.

**Target Platform**: Web application used primarily on mobile browsers (camera capture via `<input type="file" accept="image/*" capture="environment">`) and desktop browsers (file picker)

**Project Type**: Web application with separate backend API (modular monolith) and frontend single-page application

**Performance Goals**: End-to-end scan response (upload + Gemini + local DB lookup) under 5 seconds. Manual search response under 1 second. Log endpoint under 200ms.

**Constraints**: File uploads validated by magic bytes (not extension only); max upload size 10 MB; `IFoodVisionService` is the sole integration point for AI — no AI calls elsewhere; all endpoints require authentication; standard `ServiceResponse<T>` envelope on every response; no automated test projects

**Scale/Scope**: v1 scale of ~1,000 active users; 50+ seeded Egyptian food items; max 20 results returned per food search query

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### Pre-Phase-0 Gate

| Principle | Verdict | Notes |
|-----------|---------|-------|
| I. N-tier Modular Monolith (3 projects) | PASS | No new projects. Entities in `GymScan.Database`, services in `GymScan.Services`, controllers in `GymScan.API`. `IFoodVisionService` interface and Gemini implementation live in `GymScan.API/Services/` — same placement as `ICurrentUserService`. |
| II. Thin Controllers, No Logic Leakage | PASS | `FoodScanController` calls one service method per action and returns `ToActionResult()`. File byte validation happens in the service/validator layer, not the controller. |
| III. CQRS-style Service Separation (No MediatR) | PASS | `FoodScanService` handles the write path (analyze + log). `FoodSearchService` handles the read path. No MediatR. |
| IV. Standard Response Contract | PASS | All three endpoints return `ServiceResponse<T>`. All error paths return `ServiceResponse<T>.Failure(...)`. |
| V. Simplicity — No Premature Complexity | PASS | No unit tests, no test harnesses. File validation uses a simple magic-bytes byte-array check. Local DB lookup uses `Contains` string matching — no fuzzy engine. |

**Gate Result**: PASS — no violations, no Complexity Tracking entries required.

### Post-Phase-1 Recheck

Research, data model, contracts, and quickstart are fully consistent with all five constitution principles. The `IFoodVisionService` abstraction is the only pattern worth noting: it is intentionally placed at the API project level (not a new project), following the same pattern as `ICurrentUserService`.
